# Provider And Router Integration Operations

## Purpose
This document defines the operational baseline for `T-901`.

The selected deployment topology is:

- Rails control plane: Docker
- Pterodactyl Panel: Docker
- Wings: host service, not containerized
- mc-router: separate routing-plane component

## Topology Decision

Use this split unless the project explicitly changes architecture later.

- Rails app does not control Docker directly.
- Pterodactyl Panel provides provider API and administrative UI.
- Wings is the only component that needs direct Docker daemon access.
- Do not mount `/var/run/docker.sock` into the Rails app container.
- Do not mount `/var/run/docker.sock` into the Pterodactyl Panel container.
- Let Wings run on the host and manage Docker there.

## Why This Split

- It keeps Docker-control authority concentrated in Wings.
- It avoids giving Docker-daemon privileges to the Rails control plane.
- It avoids the extra complexity of running Wings with Docker-in-Docker.
- It keeps the Rails app aligned with the existing architecture rule that execution control happens through external provider APIs only.

## Component Boundaries

### Rails control plane

- Runs in this repository's Docker Compose stack.
- Calls Pterodactyl Application API for create/delete and metadata operations.
- Calls Pterodactyl Client API for lifecycle and status operations.
- Writes mc-router route config through the router integration already documented elsewhere.

Required env:

- `EXECUTION_PROVIDER_PANEL_URL`
- `EXECUTION_PROVIDER_APPLICATION_API_KEY`
- `EXECUTION_PROVIDER_CLIENT_API_KEY`
- `EXECUTION_PROVIDER_PROVISIONING_TEMPLATES`

### Pterodactyl Panel

- Runs in Docker.
- Owns the database-backed administrative API and web UI.
- Should be reachable from the Rails app over HTTP(S).
- Does not need Docker socket access in this project baseline.

### Wings

- Runs on the host as a service.
- Owns direct Docker daemon interaction.
- Should be registered to the Panel as the execution node for Rails-managed Minecraft servers.
- Must expose or report allocations that mc-router can reach as `backend_host:backend_port`.

### mc-router

- Runs separately from Rails.
- Consumes route config generated by Rails.
- Connects to provider backend targets using the allocation-derived `backend_host:backend_port`.

## Network Expectations

- Rails must be able to reach the Panel API endpoint configured in `EXECUTION_PROVIDER_PANEL_URL`.
- The Panel must be able to communicate with Wings according to the normal Pterodactyl control path.
- mc-router must be able to reach the backend allocation addresses reported by the provider node.
- If Wings runs behind NAT, ensure the allocation IP/alias reported to Rails is still reachable from the mc-router host.

## Provisioning Checklist

Before enabling real create/start/stop flows:

1. Bring up the Rails stack from this repository.
2. Bring up Pterodactyl Panel in Docker.
3. Install and start Wings on the host.
4. Register the Wings node and allocations in Panel.
5. Create or identify the service account/API keys for Rails.
6. Set `EXECUTION_PROVIDER_PANEL_URL` to the Panel API origin.
7. Set `EXECUTION_PROVIDER_APPLICATION_API_KEY`.
8. Set `EXECUTION_PROVIDER_CLIENT_API_KEY`.
9. Set `EXECUTION_PROVIDER_PROVISIONING_TEMPLATES` to match the template kinds exposed by Rails.
10. Restart Rails so the updated provider env is loaded.
11. Verify `/servers/new` shows only the intended templates.
12. Run a create request and confirm provider identifiers, backend allocation, and route publication all persist correctly.

## Anti-Patterns To Avoid

- Running Rails with Docker socket access.
- Running Panel with Docker socket access.
- Letting Rails infer lifecycle availability when `provider_server_identifier` is missing.
- Exposing template kinds in Rails that are absent from `EXECUTION_PROVIDER_PROVISIONING_TEMPLATES`.
- Switching Wings to a containerized Docker-in-Docker topology without revisiting allocation reachability and host-security assumptions.
